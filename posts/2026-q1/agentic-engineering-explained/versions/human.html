The conventional framing — “AI writes code faster” — treats an LLM-backed agent as a typing-speed multiplier. That framing predicts productivity gains proportional to output volume, and the production data does not bear it out. Teams that run agents as pure throughput tools see short-horizon wins and long-horizon decay: regressions slip in, style drifts, context gets lost across sessions, and the review bottleneck becomes the limiting factor.

Agentic engineering treats the agent as a cognitive collaborator with a well-characterized set of failure modes — primarily context rot, tool-use errors, and overconfident generation — and builds the discipline around those failures. The discipline is not interesting in the abstract. It is interesting when measured against specific metrics: time-to-first-correct-patch on benchmarks like SWE-bench, regression rate per merged PR, and the ratio of reviewer time to agent-produced lines of code.

What the agent is actually bad at

Three failure modes dominate.

Context management. An agent’s working memory is the prompt. Everything outside the prompt is invisible. Repositories of non-trivial size exceed any single-model context window, so the question is not “how big is the context” but “what gets loaded, when, and by whom.” The industry has converged on a handful of patterns — retrieval-augmented generation, project-level rules, named skills, and MCP tool servers — which answer the same question in different layers of the stack.

Tool-use reliability. Given a file-read tool, an agent will occasionally read the wrong file, hallucinate contents of files it did not read, or fail to read a file at all and proceed from memory. Tool-use error rates are non-trivial on public evaluations and do not go to zero with scale. They are engineered around, not trained away — typically with structured tool schemas, post-call verification, and deterministic caches keyed on file hashes.

Calibration. An agent will frequently present a fabricated claim with the same confidence as a verified one. Calibration does not improve with model size alone; the published frontier-model calibration curves are reliable evidence of this. It improves with grounding — the agent citing the source of a claim and the system verifying the citation — and with explicit uncertainty cues in the prompt contract.

A shipping agent system addresses all three. A demo addresses none of them.

Three patterns that carry the weight

Persistent context between sessions. Every agent session starts cold unless the project gives it a floor of mandatory context. The floor is best expressed as rules — short, imperative, machine-readable documents that encode decisions the team has already made. Rules are not memory for the agent. They are memory for the team, authored in a form the agent can consume. In this codebase, each rule carries an Origin section describing the class of problem it addresses. Ninety-plus rules compound into a floor of quality no single session has to re-derive.

Parallel execution without collision. Running multiple agents on the same repository produces branch conflicts, port collisions, cache corruption, and hours of recovery. Running them in isolated git worktrees produces none of that. Each worktree gets its own working directory, its own dev server, its own database. Agents do not know about each other. The human reviewer merges approved branches from main and sends rejections back to the originating branch. This is the closest analogue to CI-style isolation available for interactive agent work.

Structured review before merge. Plain code review does not scale to machine-generated volume. A structured adversarial-review protocol does: one persona attacks the diff (security, scope creep, silent reversions from rebases), one defends, one synthesizes. The protocol is not a human-review replacement. It is the layer that catches predictable classes of defect — missing auth guards, mis-scoped PRs, fabricated APIs — before reviewer attention is spent on them.

Grounding for user-facing agents

Agents that speak directly to users have a different constraint: every claim must be verifiable at the time it is made. “Better prompting” is not a grounding strategy; it is a calibration wish. A grounding strategy looks like: every response carries source citations with cryptographic fingerprints, a verification tool lets the agent check claims against a data store before presenting them, and the transport layer preserves the fingerprint so an auditor can reconstruct provenance after the fact. This adds latency. It converts vibes-trustworthy to operationally-trustworthy, which is the only kind of trustworthy that survives contact with a regulator or a serious customer.

What to measure

Three metrics distinguish a functional agent setup from a demo:

  • Mean time between agent-introduced regressions. Rising is progress — it means the rules system is learning from each class of defect.
  • Fraction of agent-authored PRs that pass adversarial review unmodified. This reveals whether the rules cover the current problem space.
  • Reviewer time per merged line. Falling is progress; rising is a warning that the agent is generating volume the review layer cannot absorb.

If those three move in the right direction over a quarter, the discipline is working. If they do not, the discipline is pantomime, and the sensible response is to either invest in the review layer or reduce the agent’s scope of authority until the numbers recover.
