If you run infrastructure at a big company, you've already seen the failure mode: someone "just fixes it" in the console, Terraform state and reality diverge, and six months later you're doing archaeology to figure out who granted what to whom. Or two teams merge Terraform PRs in the wrong order and the plan everyone reviewed is not the plan that hits prod. Or you have a policy doc that says "all changes via Git" and zero enforcement, which is the same as not having the policy.

Guardian is a Terraform collaboration and automation tool built to sit on top of GitHub: propose in a PR, review the plan in the thread, merge when it's right, and let automation handle the boring parts of actuation. It's not magic (the README is explicit that bad config doesn't make you secure), and it's not an official Google product even though it lives in the abcxyz org. What it is, today, is a concrete way to turn "we should treat IaC like real code" from a slide into a workflow people can actually follow.

What you get that spreadsheets don't capture

Guardian is built around GitHub primitives: pull requests, comments, workflows. Terraform plans and applies (including outputs) can show up in PR comments. It can find Terraform entrypoints across the repo and focus work on what changed instead of fanning out blindly across every module. That's the kind of operational discipline enterprises claim they want — visibility, repeatability, an audit trail that isn't a Jira ticket someone closed by mistake.

For Google Cloud shops, it also includes IAM drift detection: compare what's in Terraform state to what's actually attached in org/folder/project IAM, and surface gaps in a GitHub issue so click-ops doesn't silently become permanent truth. There's statefile drift tooling and policy enforcement hooks in the same ecosystem. I'm not going to pretend every Fortune 500 runs exactly this stack; I'm saying the problems it targets are universal, and the repo is solving them in public where you can read the code.

Why "make people use it" isn't empty management talk

Big corps love saying everyone should use the blessed pipeline. Then they hire five thousand people, half of whom have root in something, and the blessed pipeline is optional because there's always a faster path when a VP is breathing down your neck. Standardizing on Guardian (or something with the same properties — PR-centric plans, automated checks, recorded applies) isn't about being mean to developers. It's about making the fast path and the safe path the same path. When production changes always leave a PR-shaped artifact, security, audit, and on-call stop playing guessing games about whether Terraform was involved.

The merge-check story is a good example of the mindset. When lots of Terraform PRs are open, a plan can go stale after other work lands; merging anyway is how you get surprises. One heavy-handed fix is "rebase constantly," which burns time even when the merged stuff didn't touch your entrypoints. Guardian documents a guardian_merge_check-style workflow using GitHub's merge_group to simulate the merge and detect whether your Terraform entrypoints actually conflict with what's now on the target branch — block only when there's real overlap, not on every unrelated doc change. That's the sort of thing you want mandated: automation that catches the scary case without ritualized friction for everything else.

The honest pitch

Employees shouldn't "use Guardian" because the name sounds cool. They should use a workflow where infrastructure changes are proposed, reviewed, and applied through automation tied to version control. If your org picks Guardian for that, you're getting battle-tested glue for Terraform + GitHub, drift detection hooks, and policy enforcement integration — with the caveats in the README taken seriously (credentials, backends, org structure still have to be sane).

If you're a leader, the metric that matters isn't how many repos install a binary. It's whether anyone can still change prod without leaving a trace. Guardian's value is that it makes the trace — PR, plan, discussion, apply — the default, which is where big companies actually wanted to be all along. They just rarely invest enough to make the default easier than the console.
