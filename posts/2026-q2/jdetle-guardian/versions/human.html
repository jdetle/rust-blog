Guardian is a Rust daemon that measures host resource pressure on developer machines running AI coding agents, and injects that measurement into the agent’s context so the agent stops issuing requests that will fail for reasons the model cannot see.

The motivating observation is straightforward: a coding agent on a 16 GB laptop is not the same system as a coding agent on a 96 GB workstation, and the model cannot tell which one it is running on. When the host is under memory pressure, completions time out, file reads block, and the IDE stutters. The agent interprets these as transient failures and retries, which makes the pressure worse. A well-behaved agent would slow down. Nothing in the model or the editor currently tells it to.

What it does

guardiand samples a small set of host signals on a five-second interval:

  • CPU pressure and thermal state (via macOS sysctl and IOKit power metrics)
  • Resident and swap memory, per-process for Cursor and Codex
  • Disk space and inode pressure on $HOME
  • Docker Desktop footprint, when present

It writes the result atomically to ~/.guardian/state.json with a monotonic version counter. A CLI (guardian) reads the same state for status, manual snooze, and policy overrides. Editor hooks — session-start, before-submit-prompt, before-read-file, pre-tool-use, subagent-start — read state.json and inject a compact pressure summary into the agent’s context. The agent now has ground truth about the host.

Policy gates

An optional gate layer reads hook_policy.json and blocks prompt submission when a configured threshold is exceeded. The default thresholds are conservative: 80% memory, 90% swap, under 5 GB free disk. Every gate has an escape hatch: guardian proceed-once, guardian snooze <duration>, and guardian zeno (halve the applicable threshold for the next prompt, iteratively). A block is never a brick; the operator is always one command from override.

Architecture

The design choice that matters: the daemon owns state, and everything else is a reader. The CLI is a reader. The hooks are readers. There is no IPC, no server socket, no background coordination. Atomic JSON writes plus a version counter give us a consistent read model with zero operational overhead. On crash the daemon restarts and the next sample wins — there is no state worth reconciling, because any stale read is at most five seconds old.

Hooks are synchronous scripts invoked by the editor. They read state.json, format the pressure summary, and exit. The target latency budget is under 50 ms at p99 on a cold read. A disk-bounded read of a ~2 KB file sits well within that budget; caching would add failure modes without meaningful latency gain.

Why Rust

The daemon runs continuously on developer machines. Footprint and predictability matter. The Rust implementation holds under 15 MB RSS at steady state and has no GC pauses. An earlier Node prototype drifted to 80–120 MB and introduced pauses that correlated with the same pressure events the daemon exists to measure — an untenable feedback loop.

What ships

The repository contains the daemon, the CLI, Cursor and Codex hook scripts, a default config.toml with documented thresholds, a deferred-work queue for prompts blocked by gates (JSONL plus CLI), and a Dockerized stress-test harness so contributors can verify gate behaviour without saturating their own machines. A SwiftUI menu-bar app is optional and reads the same state file.

Where it fits

Guardian is useful when AI-assisted development is mandated across a fleet of standard-issue laptops. A fleet does not run on matched hardware in practice: engineers are on two-year-old MacBook Airs, brand-new MBPs, and everything in between. Mandating agents without measuring host pressure converts the fleet into a stress test. Guardian replaces “the IDE is slow today” with a legible signal and a policy that keeps prompts from piling up on a box that cannot serve them.
