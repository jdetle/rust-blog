The polyrepo-vs-monorepo debate usually gets framed as a tooling question, but it's fundamentally an organizational coupling problem. Conway's Law predicts that your system architecture will mirror your communication structure. When teams split into isolated repos, they're encoding organizational boundaries into their build graph — and those boundaries have real costs at the dependency resolution, CI/CD, and deployment layers.

Consider the concrete overhead: N repositories each maintain their own Dockerfile, .github/workflows/, tsconfig.json, linter configs, and dependency lockfiles. A security patch to a shared transitive dependency (say, a CVE in lodash or openssl) requires N separate PRs, N CI runs, N deployment cycles. The blast radius of a missed patch scales linearly with repo count. With a unified repo and a single lockfile, it's one PR, one CI run, one deploy.

The build graph is where monorepos really pay off. Tools like Turborepo, Nx, Bazel, and Buck can compute the affected dependency subgraph for any change and run only the relevant tests and builds. In a polyrepo world, you either run everything (slow) or maintain fragile heuristics for which repos to trigger (error-prone). The monorepo build cache is a strict superset of what polyrepo CI caching can achieve because it has visibility into the full dependency DAG.

The real trade-off isn't simplicity vs. complexity — it's where complexity lives. Polyrepos distribute complexity across N independent CI systems, N sets of deployment configs, and N dependency graphs that may or may not be in sync. Monorepos concentrate complexity in the build system and the VCS layer (large checkout sizes, merge contention on CODEOWNERS). The latter is tooling-solvable; the former is an ongoing operational drain.

The naming critique in the original essay is architecturally sound. "Monorepo" implies a monolith. What we're actually building is a multi-package repository with well-defined module boundaries, per-package ownership via CODEOWNERS, scoped CI via affected-graph analysis, and independent deployability. It's closer to a federated system that happens to share a VCS root than it is to a monolith.
