I build the ops layer that makes AI agents reliable in production. Not the agents themselves — the discipline around them. Rules-as-memory (every failure encoded so agents don't repeat it), adversarial review before every push (multi-persona attack/defense catching security holes and regressions), parallel agent orchestration via git worktrees, and grounding systems that prevent agents from hallucinating claims to users.

I've been doing this at PwC, at a personal project called agentdds, and in public on my blog at jdetle.com — where every post has an AI-generated version and a human version side by side, so you can see the actual difference. The rules corpus (~90 rules, each one encoding a real production failure) and the skills library (adversarial review, blog pipeline, finish-work-merge-ci, and others) are all public in the repo.

I work in three modes. Advisory (hourly or retainer) — review what you're building, identify where it'll break, explain the fix. Pair-build (weeks) — I embed, we ship something real together, I leave behind rules and scripts your team can build on. Install-the-system (fixed scope) — worktrees, rules corpus, review pipeline, CI hooks, trained team, done.

If your AI initiative is stuck in demo, the problem is almost always one of: agents with no memory between sessions, no way to run agents in parallel without clobbering each other, or no review step before failures reach users. I've fixed all three, repeatedly. If your agents hallucinate in production, that's a grounding problem and it's fixable. If you want your team to actually adopt agentic engineering (not just use Copilot, but build the system around it), that's the install-the-system engagement.
