The first time I watched an agent catch one of its own bugs — not because I told it to, but because an adversarial review step I'd wired up flagged the diff before it could push — I thought: this is what code review was supposed to be. Not the performance of code review. Actual review, done automatically, every time, before it matters.

That's the thing that's hard to explain about the work I've been doing for the last year. It's not "using AI to write code faster." That framing is technically true and almost entirely useless. What I've actually been doing is figuring out how to make AI agents reliable enough to operate in a real codebase — and the path to that ran straight through a realization about myself.

The breakthrough

I'm a jack of all trades. I've spent my career being decent at a lot of things and elite at none of them — frontend, backend, infrastructure, data, a little ML, enough Rust to be dangerous, enough React to ship production UIs. Classic generalist. For most of my career, that was a ceiling. Specialists go deeper, and depth is usually what gets rewarded.

What changed is that agents don't actually care whether you're the deepest Rust person in the room. They care whether the constraints, tradeoffs, and failure modes you've learned across disciplines are written down somewhere they can load into context. The specialist carries their depth in their head. The generalist, if they're disciplined about encoding what they've learned, can carry breadth in the rules corpus.

That's the unlock. Every hard-won lesson — the "don't do X because Y will happen" stuff that lives in a senior engineer's head and walks out the door when they leave — becomes a rule file or a skill. Once it's encoded, the agent applies it with a consistency and coverage I can't match alone. I don't have to be the world's most pedantic React reviewer; I just have to have written down what I know about React failure modes once. The agent reads it every session.

The compounding effect is real. Breadth times encoded depth turns out to be something a single specialist can't easily replicate. The generalist becomes the person who can move coherently across stacks because the discipline travels with them — not in their memory, but in the corpus.

This is also what I mean when I use the phrase "agentic engineering" instead of "vibe coding." Vibe coding is shipping something that works until it doesn't. No memory between sessions, no review step, no institutional knowledge — just prompting and hoping. Agentic engineering is building the discipline around the agent: the rules, the review protocols, the orchestration patterns, the grounding systems. The rest of this post is the four load-bearing pieces of that discipline.

Rules-as-memory

When an agent makes a mistake (and they do, constantly), the question is whether it'll make that same mistake again tomorrow. Out of the box, the answer is yes. The agent has no memory of the failure. So I built a system: every bug that makes it past the editor and bites us in CI becomes a rule. The rule lives in the codebase, gets loaded into the agent's context on every session, and explains — in plain language — what the failure was, what caused it, and what the correct pattern is. Each rule includes a "Detection Acceleration" section: how long did this take to find before the rule existed, vs. how long now?

Ninety-odd rules later, the system works. Not because AI got smarter. Because the failures are encoded and the agent reads them. For a generalist, this is the single biggest lever: each rule is a specialist's lesson I don't have to personally re-derive every time I touch a new corner of the stack. There's a whole post about this if you want the details — the short version is that the rules corpus is the most defensible part of any agentic setup I've seen. It's institutional memory that doesn't depend on who's currently on the team.

Parallel agents, no collisions

I run multiple agents simultaneously on different branches using git worktrees — each one gets its own working directory, its own dev server port, its own DB connection. They don't know about each other. Each thinks it's the only one. Meanwhile I'm reviewing work from main, merging what passes adversarial review, sending the rest back with notes. The throughput is real. Not artificial.

The setup takes about an hour to configure for a new project. The scripts are in this repo if you want to look at them. Setup scripts, branch conventions, hook files — all of it is documented, reproducible, and transferable. I've installed it in multiple codebases now. It works on any project that uses git.

Adversarial review

Before any non-trivial change goes out, I run a structured review: one persona plays offense (security holes, race conditions, scope creep, accidental reversions from rebases), one plays defense (explaining the reasoning), one synthesizes. The personas are written to be disagreeable by default. They've caught things. Real things. SSRF vulnerabilities in fetch calls I thought were fine. Missing auth guards on routes that exposed user emails. Silent error swallowing masking database failures as empty 200 responses.

The review runs as a skill — a structured prompt I can invoke before pushing. It's not foolproof. But it's consistently found things that would've made it to production otherwise. The persona framework is documented under docs/adversarial-review/ if you want to see how it's structured.

Grounding for production agents

If you're shipping an agent that makes claims to users, hallucination prevention is a different problem than most people think. The usual approach is "better prompting" or "stricter system prompts." That doesn't work reliably. What works is grounding: every response gets HMAC-fingerprinted, tagged with source citations, and a verification tool that lets the agent check its own claims against the database before presenting them. It slows things down slightly. It makes the system meaningfully trustworthy instead of vibes-trustworthy.

I built this for an MCP server that serves skills and rules data to AI agents. The agent can call a verification tool before saying "skill X is in category Y" — it checks, confirms, then says it. The fingerprint lets you audit later whether data was tampered with between retrieval and display. It's the kind of thing that seems excessive until you've watched an agent confidently present fabricated information to a senior stakeholder.
