grug work at many company over long program life. every company, grug see same thing: many repo everywhere. each repo have same dockerfile, same CI, same everything. grug ask why? nobody know. "team want own repo" they say. grug reach for club but stay calm.

problem with many repo is bug fix. grug find bug in one repo, fix. but same bug in twelve other repo! grug fix one, forget other eleven. six month later, production go boom at 2am, grug get page. same bug! grug very frustrate.

some big brain say "monorepo solve this!" and grug think maybe big brain right for once. put all code in one place, fix bug once, all team get fix. sound good to grug brain.

but then grug learn truth: problem not really about where code live. problem about how team think about code. team want wall around code because team not trust other team. is organization problem, not git problem.

grug think word "monorepo" was mistake. sound scary, sound like monolith. manager hear "mono" and run away. instead should call "multi-package repo" because that what actually is: many small thing in one place with shared tool. like grug tool shed — many club, one shed, each club for different dinosaur.

complexity demon love when team split code into many repo. demon get stronger with each new repo. grug say: keep repo count low, keep complexity demon weak. is simple advice but grug find most important advice is simple.
